# Mindmap 프로젝트 계획서

작성일: 2026-06-11
상태: 계획 단계

## 1. 프로젝트 한 줄 정의

Tauri 기반 데스크톱 마인드맵 프로그램. 하나의 마인드맵은 하나의 Markdown 파일로 저장하며, 마우스보다 키보드 중심 편집을 지향한다.

## 2. 해결하려는 문제

핵심 문제:

- 일반 마인드맵 도구는 마우스 조작 비중이 커서 생각을 빠르게 입력하고 재구성하기 어렵다.
- 시각 마인드맵 데이터가 독자 포맷에 갇히면 장기 보관과 다른 도구 연동이 불편하다.
- Markdown 메모처럼 파일 단위로 소유하고 관리할 수 있으면서도, 편집할 때는 시각적 마인드맵으로 다루고 싶다.

제품 방향:

- Markdown 파일을 원본 데이터로 삼는다.
- 앱은 Markdown을 마인드맵 편집 화면으로 해석하고 다시 Markdown으로 저장한다.
- GitHub에서 자연스럽게 렌더링되는 Markdown 표준 구조를 우선하며, 비표준 확장 문법은 지양한다.
- Markdown에는 문서의 의미와 구조만 저장한다.
- 앱 상태, 편의 데이터, 히스토리, 설정, 캐시는 SQLite에 저장한다.
- 배치는 Markdown 구조와 순서에서 결정적으로 자동 계산하며, 좌표를 파일에 저장하지 않는다.
- 마인드맵으로 해석할 수 없는 Markdown은 조용히 실패하지 않고, 원인과 수정 방법을 자세히 진단한다.
- 예측 가능한 사용자 경험을 위해 자동 복구는 지양한다.
- 사용자는 가능한 한 키보드만으로 노드를 만들고, 이동하고, 구조를 바꿀 수 있어야 한다.

## 3. 대상 사용자

초기 가정:

- 개인 사용자가 중심이다.
- 빠른 아이디어 정리와 재구성이 중요하다.
- 복잡한 협업 기능보다 키보드 편집감, 로컬 파일 소유권, Markdown 호환성이 우선이다.

확정된 방향:

- Markdown은 외부 에디터에서 읽고 편집 가능한 원본 문서로 유지한다.
- 앱 내부에는 raw Markdown 편집 모드를 제공하지 않는다. 원문 편집은 VS Code, Obsidian 같은 외부 에디터의 역할로 둔다.
- 접힘, 선택, 커서, 줌, 팬, 노드 크기 같은 정보는 문서 의미가 아니므로 Markdown에 저장하지 않는다.
- 첫 버전은 단일 파일 열기/편집 흐름에 집중한다.

## 4. 핵심 사용자 흐름

첫 버전 후보:

1. 새 Markdown 마인드맵 파일을 만든다.
2. 루트 노드를 작성한다.
3. 키보드로 자식/형제 노드를 빠르게 추가한다.
4. 키보드로 선택 노드를 이동하고 필요한 형제/자식 노드를 생성한다.
5. 변경 내용은 자동 저장된다.
6. 결과를 `.md` 파일로 다시 연다.

보조 흐름 후보:

- 기존 Markdown 파일 열기
- Markdown 파일 다른 이름으로 저장
- 검색하기
- 키보드 명령 팔레트
- 링크, 강조, 인라인 코드 같은 Markdown 인라인 문법 편집
- 외부 diff 도구로 앱 상태와 디스크 상태 비교

## 5. MVP 범위

반드시 포함할 후보:

- Tauri 데스크톱 앱
- Markdown 파일 열기/저장
- 지연 자동 저장과 원자적 저장
- 외부 변경 감지와 자동 리로드/진단/충돌 처리
- 결정적 자동 배치 기반 마인드맵 표시
- 노드 추가, 수정, 삭제
- 부모-자식 관계 편집
- 키보드 중심 선택/편집/구조 변경
- 세션 내 undo/redo
- SQLite 기반 앱 상태 저장소
- 하나의 마인드맵을 하나의 `.md` 파일로 저장

첫 버전에서 미룰 후보:

- raw Markdown 내부 편집 모드
- 노드 내부 실제 줄바꿈
- 노드별 크기 조절
- 상시 노출되는 노드별 마우스 액션 버튼
- 접기/펼치기
- 영속 undo/redo
- 실시간 협업
- 계정 시스템
- 서버 동기화
- 복잡한 권한 관리
- AI 자동 생성
- 플러그인 시스템
- 모바일 지원

## 6. 기술 스택 후보

확정/후보:

- Tauri
- React + TypeScript + Vite
- 상태 관리: Zustand 또는 React 내장 상태
- 그래프/캔버스: React Flow, SVG 직접 렌더링, Canvas 기반 구현 중 선택
- 저장: Markdown 파일 기반 저장
- 앱 상태 저장소: SQLite

선택 기준:

- 레이아웃과 인터랙션 확장이 쉬운가?

- Markdown 원본과 앱 내부 모델 사이의 왕복 변환이 안정적인가?
- GitHub Markdown에서 읽기 좋은 파일을 유지하는가?
- 데스크톱 앱에서 로컬 파일 접근과 저장 흐름이 자연스러운가?

## 7. Markdown 저장 포맷 초안

원칙:

- 하나의 `.md` 파일이 하나의 마인드맵이다.
- MVP에서는 파일 전체를 하나의 마인드맵 문서로 취급한다.
- 앱은 마인드맵 영역만 부분 파싱하고 나머지 Markdown을 보존하는 모드를 제공하지 않는다.
- GitHub에서 자연스럽게 렌더링되는 Markdown 구조를 우선한다.
- 파일은 먼저 CommonMark/GFM 계열 Markdown 파서로 파싱되어야 한다.
- 앱은 Markdown AST를 만든 뒤, 그 위에서 마인드맵 subset 규칙을 검증한다.
- 유효한 Markdown이어도 마인드맵 subset 규칙을 위반하면 마인드맵 문서로 열지 않고 진단한다.
- Markdown 문법 자체를 앱 전용 줄 파서나 정규식으로 재해석하지 않는다.
- 비표준 확장 문법, frontmatter, 앱 전용 HTML comment 메타데이터는 지양한다.
- 사람이 일반 Markdown 편집기로 읽고 수정할 수 있어야 한다.
- 첫 버전에서는 좌표, 접힘 상태, 선택 상태 같은 앱 전용 뷰 상태를 파일에 저장하지 않는다.
- 배치는 저장 데이터가 아니라 렌더링 결과이며, Markdown 구조와 순서만으로 계산한다.
- 노드 ID, UUID, HTML comment ID, 앱 전용 속성은 Markdown에 저장하지 않는다.
- 노드 텍스트는 데이터상 단일 라인이다. 화면에서는 자동 줄바꿈할 수 있지만 실제 줄바꿈은 저장하지 않는다.
- 앱은 Markdown 원문 편집기가 아니라 구조 편집기이며, 앱 내부 편집으로 파싱 불가능한 Markdown을 만들지 않는다.

최소 유효 구조:

```md
# Example Mindmap

- Root idea
  - Child idea
  - Another child
    - Nested idea
```

좌우 배치가 필요한 구조:

```md
# Example Mindmap

## Right

- Explicit right idea
  - Right child

## Left

- Left idea
  - Left child
```

해석 규칙:

- 첫 번째 H1을 마인드맵 루트 노드 텍스트로 사용한다.
- H1 제목은 Markdown parser가 해석한 heading content를 따른다.
- 빈 H1은 Markdown parser가 H1으로 해석하면 빈 루트 제목으로 허용한다.
- MVP에서는 heading도 앱의 canonical Markdown subset만 허용한다.
- H1은 `# Title` 또는 빈 제목의 `#` 형식만 허용한다.
- H1 marker 뒤에는 제목이 있을 때 정확히 공백 1칸만 허용한다.
- H1 closing `#` sequence는 MVP에서 허용하지 않는다.
- H1 marker 주변 과도한 공백이나 closing sequence는 자동 정리하지 않고 진단 오류로 처리한다.
- H1 다음에는 빈 줄 1개가 있어야 한다.
- 첫 번째 의미 있는 블록은 H1이어야 한다.
- H1 이전의 YAML frontmatter, 문단, 목록, 코드블록, 표, blockquote 등은 허용하지 않는다.
- H1은 필수다. 없으면 `MM001` 오류로 처리한다.
- H1은 정확히 하나만 허용한다.
- 두 번째 H1이 나오면 하나의 파일에 여러 마인드맵이 있는 것으로 보고 진단 오류로 처리한다.
- 앱의 문서/창/탭 제목은 Obsidian처럼 파일명을 기본으로 표시한다.
- H1은 파일 이름이 아니라 마인드맵 문서 안의 루트 heading이다.
- 파일명과 H1이 달라도 경고하거나 자동으로 맞추지 않는다.
- 파일명 변경과 H1 편집은 서로 독립적인 동작이다.
- H1 아래에는 최소 하나의 최상위 목록 항목이 필요하다. 없으면 `MM002` 오류로 처리한다.
- H1 바로 아래의 일반 최상위 목록은 방향 섹션이 없는 단순 오른쪽 마인드맵일 때만 허용한다.
- 방향 섹션 heading은 정확히 `## Right` 또는 `## Left`만 허용한다.
- 방향 섹션 marker 뒤에는 정확히 공백 1칸만 허용한다.
- 방향 섹션명은 대소문자를 구분하며, 별칭이나 번역어를 허용하지 않는다.
- 방향 섹션 closing `#` sequence는 MVP에서 허용하지 않는다.
- `## right`, `## LEFT`, `##Right`, `##  Right`, `## Left #`, `## Left side` 같은 heading은 자동 보정하지 않고 진단 오류로 처리한다.
- 방향 섹션 다음에는 빈 줄 1개가 있어야 한다.
- `## Right`와 `## Left`는 각각 최대 한 번만 허용한다.
- 중복 방향 섹션은 자동 병합하지 않고 진단 오류로 처리한다.
- 빈 `## Right` 또는 `## Left` 섹션은 허용한다.
- 단, H1 아래 전체 마인드맵에는 최소 하나의 목록 항목이 있어야 한다.
- `## Left` 또는 `## Right` 섹션을 하나라도 사용하면, 루트의 직접 자식은 모두 방향 섹션 아래에 있어야 한다.
- `## Left` 섹션 아래의 최상위 목록은 루트의 왼쪽 자식으로 해석한다.
- `## Right` 섹션 아래의 최상위 목록은 루트의 오른쪽 자식으로 해석한다.
- `## Right`와 `## Left`의 문서 내 순서는 의미에 영향을 주지 않는다.
- 좌우 방향은 루트의 직접 자식에만 적용한다.
- 루트 직접 자식 아래의 모든 하위 노드는 부모 방향을 상속하며, 별도의 좌우 분기 규칙을 두지 않는다.
- `-` marker를 사용한 중첩 unordered list를 부모-자식 관계로 해석한다.
- `*` 또는 `+` marker를 사용한 unordered list는 MVP에서 지원하지 않는다. 마인드맵 영역 안에서는 진단 오류로 처리하고 `-` marker로 바꾸도록 안내한다.
- ordered list는 MVP에서 지원하지 않는다. 마인드맵 영역 안의 ordered list는 진단 오류로 처리하고 `-` marker의 unordered list로 바꾸도록 안내한다.
- 목록 들여쓰기는 depth당 공백 2칸만 허용한다.
- 루트 직접 자식은 들여쓰기 0칸, 자식은 부모보다 정확히 2칸 더 들여쓴다.
- 탭 문자, 1칸/3칸 들여쓰기, depth와 맞지 않는 4칸 들여쓰기는 진단 오류로 처리한다.
- 목록 항목 사이에는 빈 줄을 허용하지 않는다.
- heading과 목록 사이, 방향 섹션 사이의 빈 줄은 앱의 canonical 포맷과 정확히 일치해야 하며, 과도하거나 누락된 빈 줄은 진단 오류로 처리한다.
- 줄 끝은 LF만 허용한다.
- CRLF는 MVP에서 진단 오류로 처리한다.
- 파일은 마지막 개행 1개로 끝나야 한다.
- 파일 끝의 추가 빈 줄은 허용하지 않는다.
- heading, 빈 줄 같은 구조 줄의 trailing spaces는 허용하지 않고 진단 오류로 처리한다.
- list item 줄의 trailing spaces는 노드 텍스트 일부로 보존한다.
- 빈 목록 항목은 `-`로 쓰며, `- `처럼 구분 공백만 있고 텍스트가 없는 목록 항목은 canonical 형식이 아니므로 진단 오류로 처리한다.
- 목록 항목 안의 링크, 강조, 인라인 코드 등 GitHub Markdown 기본 인라인 문법은 노드 텍스트로 보존한다.
- MVP에서는 노드 텍스트 안의 인라인 Markdown을 원문 문자열로 보존하고 plain text로 편집한다.
- 앱은 MVP에서 인라인 Markdown을 WYSIWYG 렌더링하거나 구조화하지 않는다.
- 링크 클릭, 강조 렌더링, 인라인 코드 렌더링, preview mode는 후순위 기능으로 둔다.
- 목록 항목은 단일 라인 노드 텍스트로 해석한다.
- 노드 텍스트는 단일 라인 Unicode 문자열이다.
- 한글, 영어, 숫자, 일반 기호, emoji는 허용한다.
- Markdown escape와 HTML entity는 해석하지 않고 원문 문자열로 보존한다.
- 실제 newline, tab, NUL, BEL 등 제어 문자는 노드 텍스트 안에서 허용하지 않는다.
- zero-width character는 MVP에서 허용하되, 나중에 정규화 기능에서 감지/정리할 수 있다.
- 빈 목록 항목은 빈 텍스트를 가진 유효한 노드로 해석한다.
- 앱은 빈 노드를 읽기/저장 과정에서 자동 삭제하거나 건너뛰지 않는다.
- 단, 편집 중 정확히 빈 텍스트인 일반 노드에서 `Esc`를 누르면 사용자가 해당 빈 노드를 취소한 것으로 보고 삭제한다.
- 텍스트와 자식이 모두 없는 빈 리프 노드는 포커스를 잃으면 사용자가 생성을 취소한 것으로 보고 삭제한다.
- list marker 뒤의 구분 공백 1칸을 제외한 나머지 텍스트는 그대로 노드 텍스트로 보존한다.
- 앱은 노드 텍스트의 앞뒤 공백을 자동 trim하지 않는다.
- 공백만 있는 목록 항목은 공백 텍스트를 가진 유효한 노드로 해석한다.
- 공백 정리, 빈 노드 삭제, marker/들여쓰기 정리 같은 정규화는 사용자가 명시적으로 실행하는 별도 기능으로만 제공한다.
- Markdown list item continuation line, 여러 문단을 가진 list item, 노드 내부 실제 줄바꿈은 MVP에서 지원하지 않는다.
- 사용자가 `\n`을 입력하면 줄바꿈 명령이 아니라 일반 텍스트로 취급한다.
- H1 이후에는 마인드맵 구조에 필요한 `## Right`, `## Left`, `-` 목록만 허용한다.
- 그 밖의 heading, 문단, 표, 코드블록, blockquote 등은 마인드맵 영역 안의 지원하지 않는 구조로 진단한다.
- 앱은 읽은 구조를 암묵적으로 다른 구조로 정규화하지 않는다.
- 구조 변경이 필요한 경우에는 진단을 보여주고, 나중에 명시적 정규화 옵션으로 처리한다.
- 파일이 위 구조와 맞지 않으면 임의로 변환하지 않고 진단 결과를 보여준다.

검토 포인트:

- 노드 ID는 파일에 저장하지 않고, 로드 시 구조 순서 기반으로 임시 생성한다.
- 같은 텍스트의 노드가 여러 개 있을 수 있으므로 런타임 ID는 저장 세션 안에서만 안정적이면 된다.
- SQLite에서 노드를 참조해야 할 때도 구조 경로 기반 임시 ID를 우선 사용한다.
- 외부 편집이나 구조 변경으로 구조 경로가 맞지 않으면 선택, 커서, 접힘, 노드 크기 같은 앱 상태 복원을 포기한다. 이는 오류가 아니다.
- 일반 Markdown 편집기로 수정한 뒤 문제가 생기면, 가능한 한 정확한 진단과 수정 방법을 제공해야 한다.

## 8. 읽기 실패 진단 원칙

목표:

- 사용자가 파일을 왜 마인드맵으로 읽을 수 없는지 즉시 이해할 수 있어야 한다.
- Rust 컴파일러처럼 원인, 위치, 기대한 구조, 수정 제안을 함께 제공한다.
- 자동 복구보다 정확한 실패 원인과 수정 가이드를 우선한다.
- 앱은 사용자가 명령하기 전까지 문서를 정리하거나 고치지 않는다.

진단 메시지에 포함할 정보:

- 오류 코드: 예를 들어 `MM001`
- 심각도: `error`, `warning`, `info`
- 파일 경로
- 줄/열 위치
- 문제가 된 Markdown 일부
- 앱이 기대한 구조
- 실제 발견한 구조
- 수정 제안
- 자동 수정 가능 여부. MVP에서는 안내만 제공하고 실행 옵션은 후순위로 둔다.

예시:

```text
error[MM001]: mindmap root heading is missing
  --> notes.md:1:1
   |
 1 | - Topic A
   | ^ expected a level-1 heading before the root list
   |
   = expected: # My Mindmap
   = help: add a first-level heading at the top of the file
```

주요 진단 후보:

- `MM001`: 첫 번째 H1이 없음
- `MM002`: H1 아래에서 최상위 목록을 찾을 수 없음
- `MM003`: 목록 들여쓰기가 depth당 공백 2칸 규칙을 위반함
- `MM004`: 목록 항목 안에 블록 요소가 있어 노드 텍스트로 해석할 수 없음
- `MM005`: 여러 개의 독립 최상위 목록 중 어떤 것을 마인드맵으로 쓸지 모호함
- `MM006`: 지원하지 않는 Markdown 구조가 마인드맵 영역 안에 있음
- `MM007`: 방향 섹션과 H1 바로 아래 일반 목록이 섞여 있음
- `MM008`: 노드 내부 멀티라인 또는 list item continuation line이 있음
- `MM009`: ordered list가 마인드맵 영역 안에 있음
- `MM010`: `*` 또는 `+` list marker가 마인드맵 영역 안에 있음
- `MM011`: 탭 문자가 목록 들여쓰기에 사용됨
- `MM012`: H1 이전에 지원하지 않는 Markdown 블록이 있음
- `MM013`: H1 이후 마인드맵 영역에 지원하지 않는 Markdown 블록이 있음
- `MM014`: 방향 섹션 heading이 정확한 `## Right` 또는 `## Left`가 아님
- `MM015`: `## Right` 또는 `## Left` 방향 섹션이 중복됨
- `MM016`: heading이 앱의 canonical heading 형식을 위반함
- `MM017`: 빈 줄이 앱의 canonical 포맷과 일치하지 않음
- `MM018`: 구조 줄 끝 또는 파일 끝 형식이 canonical 포맷과 일치하지 않음
- `MM019`: 노드 텍스트에 허용하지 않는 제어 문자가 있음
- `MM020`: H1이 두 개 이상 있음

처리 정책:

- `error`: 마인드맵으로 열지 않고 진단 화면을 보여준다.
- `warning`: 마인드맵으로 열되, 손실되거나 무시된 구조를 알려준다.
- `info`: 정상 처리했지만 정규화 시 바뀔 내용을 알려준다.
- MVP에서는 자동 수정을 제공하지 않는다.
- 나중에 정규화/자동 수정 옵션을 추가하더라도, 오류 내용과 수정 방안을 먼저 보여주고 사용자가 명시적으로 선택해야 한다.
- 정규화 기능은 사용자가 명령했을 때 한 번에 실행되는 별도 명령이어야 하며, 읽기/저장 과정에서 암묵적으로 실행하지 않는다.
- 읽을 때 허용한 문서를 저장 시 자동으로 다른 구조로 바꾸지 않는다.

## 9. 데이터 모델 초안

저장 책임:

- Markdown은 문서 의미와 구조의 유일한 원본이다.
- SQLite는 앱 상태와 편의 데이터를 저장하는 보조 저장소다.
- SQLite가 없거나 깨져도 `.md` 파일만으로 문서는 열리고 편집 가능해야 한다.
- SQLite 데이터는 잃어버려도 문서 의미가 손상되지 않는 정보만 담는다.

```ts
type Mindmap = {
  id: string;
  title: string;
  rootNodeId: string;
  nodes: Record<string, MindmapNode>;
  edges: MindmapEdge[];
  sourcePath?: string;
  sourceHash?: string;
};

type MindmapNode = {
  id: string;
  path: string;
  text: string;
  direction?: "left" | "right";
  parentId?: string;
  childrenIds: string[];
};

type MindmapViewState = {
  selectedNodePath?: string;
  editingNodePath?: string;
  cursorOffset?: number;
  zoom?: number;
  pan?: { x: number; y: number };
};

type MindmapEdge = {
  id: string;
  sourceId: string;
  targetId: string;
};
```

검토 포인트:

- 첫 버전은 트리 구조를 우선할지, 여러 부모나 임의 그래프도 허용할지
- 자동 레이아웃을 기본으로 하고 수동 위치 조정은 지원하지 않을지
- Markdown에 저장하지 않는 런타임 상태와 저장할 문서 상태를 어떻게 분리할지

SQLite에 저장할 수 있는 앱 데이터:

- 최근 파일 목록
- 파일 경로와 기준 해시/mtime
- 선택 노드, 편집 중인 노드, 커서 위치
- 줌/팬 위치
- 앱 설정
- 외부 diff 툴 설정
- undo/redo 히스토리
- 접힘 상태
- 나중에 추가할 노드별 폭/크기
- 캐시와 진단 정보

SQLite에 저장하지 않는 문서 데이터:

- 제목
- 노드 텍스트
- 노드 계층
- 목록 순서
- `## Right` / `## Left` 방향 정보

영속 히스토리 정책:

- MVP에서는 세션 내 undo/redo만 지원한다.
- 나중에 영속 히스토리를 추가할 때도 Markdown에는 저장하지 않고 SQLite에만 저장한다.
- 히스토리는 기준 파일 해시가 맞을 때만 신뢰한다.
- 외부 변경으로 기준 해시가 달라지면 기존 히스토리는 그대로 적용하지 않고 새 기준점에서 시작한다.
- 히스토리 로드 실패는 오류가 아니며, 새 히스토리로 시작한다.

## 10. 자동 배치 원칙 초안

목표:

- 같은 Markdown 파일은 항상 같은 구조와 순서로 렌더링한다.
- 사용자가 좌표를 직접 관리하지 않아도 예쁘고 예측 가능한 배치를 제공한다.
- 구조 변경은 Markdown 목록의 순서와 들여쓰기 변경으로 표현한다.

배치 규칙:

- 첫 번째 H1을 중앙 루트로 둔다.
- H1 아래 일반 최상위 목록 항목은 방향 섹션이 없는 단순 오른쪽 마인드맵일 때만 오른쪽 직접 자식으로 둔다.
- `## Left` 아래 최상위 목록 항목은 루트의 왼쪽 직접 자식으로 둔다.
- `## Right` 아래 최상위 목록 항목은 루트의 오른쪽 직접 자식으로 둔다.
- `## Left`와 `## Right`가 없으면 모든 직접 자식은 오른쪽에 배치한다.
- 방향 섹션이 있는 문서와 H1 바로 아래 일반 목록은 섞지 않는다. 섞여 있으면 `MM007` 오류로 처리한다.
- `## Right`와 `## Left` 섹션의 순서는 렌더링 결과에 영향을 주지 않는다.
- 각 하위 트리는 부모가 배치된 방향을 따른다. 루트의 직접 자식 이후에는 좌우를 다시 나누지 않는다.
- 형제 노드의 세로 순서는 Markdown 목록 순서를 따른다.
- 노드 간 간격은 하위 트리 크기를 기준으로 자동 계산한다.
- 노드가 추가/삭제/이동되면 전체 레이아웃을 다시 계산한다.
- 노드 좌표는 Markdown에 저장하지 않는다.
- 노드 좌표는 MVP에서 SQLite에도 저장하지 않는다.
- 줌/팬 위치는 SQLite 앱 상태로 저장할 수 있다.
- 노드 텍스트는 데이터상 단일 라인이지만, 화면에서는 앱이 정한 최대 폭에서 자동 줄바꿈한다.
- 시각적 줄바꿈 위치는 저장하지 않는다.
- MVP에서는 노드별 폭/높이 조절을 제공하지 않는다.
- 나중에 노드별 크기 조절을 추가한다면 SQLite 앱 상태로만 저장하고, 구조 변경으로 잃어버릴 수 있음을 허용한다.

검토 포인트:

- 나중에 명시적 정규화 옵션을 추가할 때 정규화 결과와 실행 여부를 사용자에게 어떻게 확인받을지
- 선택 노드가 화면 밖으로 밀리지 않도록 자동 스크롤/팬을 적용할지
- 아주 큰 문서에서 전체 재계산이 충분히 빠른지

## 11. 키보드 편집 모델 초안

목표:

- 생각을 입력하는 흐름을 끊지 않는다.
- 노드 생성, 이동, 편집, 구조 변경을 키보드로 수행한다.
- 앱 내부 편집 명령은 항상 유효한 mindmap AST를 만든다.
- 마우스는 보조 수단으로 둔다.

명령 후보:

- 편집 중 `Enter`: 다음 형제 노드로 이동하고, 다음 형제가 없으면 선택 노드 다음에 형제 노드를 추가한다.
- 편집 중 `Shift+Enter`: 위 형제 노드가 있으면 이동하고, 없으면 선택 노드 위에 형제 노드를 추가한다. MVP에서는 노드 내부 줄바꿈을 만들지 않는다.
- 편집 중 `Tab`: 첫 자식 노드로 이동하고, 자식이 없으면 선택 노드 아래에 자식 노드를 추가한다. 실제 탭 문자를 입력하지 않고 구조 변경 명령으로 동작한다.
- 편집 중 `Shift+Tab`: 부모 노드로 포커스/선택을 이동한다.
- `ArrowUp` / `ArrowDown`: 화면상 위/아래에 있는 가장 가까운 노드 선택. 오른쪽 가지 노드는 왼쪽 상단, 왼쪽 가지 노드는 오른쪽 상단을 기준점으로 삼아 같은 시각적 열의 노드를 우선한다.
- `ArrowLeft` / `ArrowRight`: 화면상 왼쪽/오른쪽으로 한 세대씩 이동한다. 부모 또는 바로 아래 자식 세대만 거치며, 정렬된 더 먼 세대로 건너뛰지 않는다.
- 선택 모드 `Enter`: 노드 편집 시작
- `Esc`: 편집 종료 또는 선택 모드 복귀
- 마우스 클릭은 노드 선택만 수행하며, 편집 시작은 선택 모드 `Enter`로 분리한다.
- `Delete` / `Backspace`: 노드 삭제
- `Cmd+S`: pending save flush 또는 저장 상태 확인. 기본 저장 모델은 자동 저장이다.
- `Cmd+O`: Markdown 파일 열기
- `Cmd+P`: 명령 팔레트
- `Cmd+Z`: undo
- `Cmd+Shift+Z` 또는 `Cmd+Y`: redo

검토 포인트:

- Vim 스타일 이동키를 추가할지
- Enter와 Tab 동작을 아웃라이너처럼 만들지, 마인드맵 전용으로 다르게 만들지
- 텍스트 편집 중 단축키와 선택 모드 단축키를 어떻게 분리할지

확정된 편집 원칙:

- 노드 텍스트는 단일 라인이다.
- `Enter`는 항상 새 노드 생성이다.
- `Tab`은 스마트 탭처럼 현재 노드의 계층을 바꾸며, 저장된 Markdown에는 depth당 공백 2칸만 쓴다.
- IME 입력은 조합 중간 상태가 아니라 조합 확정 단위를 히스토리 단위로 삼는다.
- 일반 문자 입력과 삭제는 글자 단위까지 세밀하게 undo/redo 가능해야 한다.
- 붙여넣기는 붙여넣은 전체 텍스트 또는 노드 묶음이 하나의 undo 단위다.
- 선택/커서 이동 자체는 문서 변경이 아니므로 undo/redo 대상이 아니다.
- 텍스트 편집 명령의 undo/redo는 필요한 경우 커서를 함께 복원한다.

## 12. 저장, 외부 변경, 클립보드

자동 저장:

- 편집 명령은 즉시 앱 상태와 undo/redo 히스토리에 반영한다.
- 디스크 저장은 매 명령마다 즉시 수행하지 않고 debounce/idle 기반 지연 저장으로 배치한다.
- 저장 후보 Markdown은 저장 직전에 `mm` 파서로 검증한다.
- 검증 성공 시 원자적으로 저장한다.
- 파일 전환, 앱 종료, 창 닫기, 백그라운드 진입 전에는 pending save를 flush한다.
- 저장 성공 후 기준 해시/mtime을 갱신한다.

외부 변경 감지:

- 외부 변경 감지는 file watcher와 polling을 함께 사용할 수 있다.
- 앱 상태가 clean이고 외부 Markdown이 파싱 가능하면 자동 리로드한다.
- 자동 리로드는 undo 히스토리에 `Reload external changes` 명령으로 기록한다.
- 외부 리로드를 undo하면 앱 상태는 디스크와 달라지므로 dirty 상태가 된다.
- 앱 상태가 clean이고 외부 Markdown이 파싱 불가능하면 현재 화면을 유지하고 외부 파일 오류 진단을 보여준다.
- 앱 상태가 dirty일 때 외부 변경이 감지되면 자동 병합하지 않고 충돌 상태로 둔다.
- 충돌 상태에서는 앱의 현재 저장 후보 Markdown을 먼저 파싱 검증한다.
- 검증 실패 시 저장/덮어쓰기/diff 진행 전에 파서 진단을 보여준다.
- 검증 성공 후 사용자는 전체 파일 단위로만 선택한다.

충돌 선택지:

- 디스크 버전 다시 불러오기
- 앱 버전으로 덮어쓰기
- 앱 버전을 다른 이름으로 저장
- 외부 diff 도구로 비교

외부 diff 도구:

- 앱은 자체 병합 UI를 만들지 않는다.
- diff는 읽기 전용 참고 도구다.
- 앱의 현재 저장 후보 Markdown과 디스크 최신 Markdown을 임시 파일로 만들고, 설정된 외부 diff 명령에 전달한다.
- diff 결과를 앱이 해석하거나 자동 반영하지 않는다.
- 부분 병합, 노드 단위 병합, 자동 병합은 MVP 범위 밖이다.

클립보드:

- 클립보드는 표준 텍스트/Markdown만 사용한다.
- 앱 전용 메타데이터는 클립보드에 넣지 않는다.
- 노드 ID, SQLite 참조, 내부 경로, 접힘 상태, 선택 상태는 복사하지 않는다.
- 노드 또는 서브트리 복사는 Markdown 목록 형태로 한다.
- 텍스트 선택 복사는 일반 텍스트로 한다.
- 붙여넣기는 텍스트 또는 Markdown 목록을 파싱해 유효한 노드 트리로 변환한다.
- 붙여넣기 결과가 유효하지 않으면 적용하지 않고 진단한다.
- 붙여넣기 전체는 하나의 undo 단위다.

## 14. 주요 결정 사항

- Markdown 저장 포맷은 H1 + 중첩 목록 기반으로 한다.
- 파일 전체를 하나의 마인드맵 문서로 취급한다.
- 목록 marker는 `-`만 허용한다.
- 목록 들여쓰기는 depth당 공백 2칸만 허용한다.
- 빈 목록 항목은 유효한 빈 텍스트 노드로 허용한다.
- 노드 텍스트의 앞뒤 공백은 자동 trim하지 않고 그대로 보존한다.
- 정규화는 사용자가 명시적으로 실행하는 별도 명령으로만 제공한다.
- 인라인 Markdown은 MVP에서 원문 문자열로만 보존하고 plain text로 편집한다.
- 노드 텍스트는 단일 라인 Unicode 문자열이며, 실제 newline, tab, NUL, BEL 등 제어 문자는 허용하지 않는다.
- H1 없음은 `MM001` 오류로 처리한다.
- H1은 정확히 하나만 허용한다.
- 앱의 문서/창/탭 제목은 파일명을 기본으로 표시한다. H1은 마인드맵 루트 노드 텍스트로 취급한다.
- H1 아래 최상위 목록 없음은 `MM002` 오류로 처리한다.
- 빈 방향 섹션은 허용하되, 전체 마인드맵에 목록 항목이 하나도 없으면 `MM002` 오류로 처리한다.
- 기본 배치는 오른쪽이며, 방향이 필요한 경우 `## Left`와 `## Right` 섹션으로 명시한다.
- 방향 섹션과 H1 바로 아래 일반 목록이 섞이면 `MM007` 오류로 처리한다.
- 방향 섹션 heading은 정확히 `## Right` 또는 `## Left`만 허용한다.
- `## Right`와 `## Left`는 각각 최대 한 번만 허용한다.
- heading은 앱의 canonical 형식만 허용한다. closing `#`와 과도한 marker 뒤 공백은 진단 오류로 처리한다.
- 빈 줄은 앱의 canonical 포맷만 허용한다. heading 다음에는 빈 줄 1개를 두고, 목록 항목 사이에는 빈 줄을 두지 않는다.
- 줄 끝은 LF, 파일 끝은 마지막 개행 1개만 허용한다. 구조 줄 trailing spaces와 EOF 추가 빈 줄은 진단 오류로 처리하되, list item 줄의 trailing spaces는 노드 텍스트로 보존한다.
- 읽기와 저장이 암묵적으로 문서를 바꾸지 않는다.
- Markdown에는 노드 ID와 앱 메타데이터를 저장하지 않는다.
- 앱 관련 상태와 편의 데이터는 SQLite에 저장한다.
- SQLite 데이터는 잃어버려도 문서 의미가 손상되지 않아야 한다.
- 앱 내부에는 raw Markdown 편집 모드를 제공하지 않는다.
- 노드 내부 실제 줄바꿈은 MVP에서 지원하지 않는다.
- 노드별 크기 조절과 접기/펼치기는 MVP에서 제공하지 않는다.
- 노드 좌표는 저장하지 않고 자동 레이아웃으로 계산한다.
- 자동 저장은 지연 저장으로 처리한다.
- 외부 변경은 자동 병합하지 않는다.
- Obsidian 연동은 단순 파일 호환성인가, vault 탐색까지 포함하는가?
- Tauri 파일 접근은 단일 파일 열기/저장부터 시작할 것인가, 작업 폴더 개념을 둘 것인가?
- AI 기능은 초기 범위에 포함하는가?

## 16. 구현 현황: 2026-06-12

- 렌더링은 MVP에서 React DOM 기반 직접 렌더링으로 시작했다. React Flow, Canvas, SVG 전용 렌더러는 아직 도입하지 않았다.
- 오른쪽 가지는 부모 오른쪽으로, 왼쪽 가지는 부모 왼쪽으로 자식 서브트리가 펼쳐지도록 기본 자동 배치를 구현했다.
- 루트와 1단계 자식 사이에도 연결선을 구현했다. 루트에서 좌/우 브랜치 rail로 이어지고, rail에서 각 root child로 선이 이어진다.
- Markdown parser/serializer와 진단 코드를 TypeScript 코어로 구현했다.
- 구조 편집 명령을 `src/core/tree.ts`로 분리했다: root/child/sibling 추가, 텍스트 수정, 삭제, 들여쓰기, 내어쓰기, 형제 순서 이동.
- 히스토리는 `src/core/history.ts`에 세션 내 undo/redo로 구현했다. 일반 문자 입력도 각 입력 이벤트가 별도 undo 단위가 된다.
- 문서 상태 머신은 `src/core/document.ts`에 구현했다: clean/dirty, pending/saving/error, clean 외부 변경 자동 리로드, invalid 외부 파일 진단, dirty 충돌 처리.
- Tauri command로 Markdown 파일 읽기, 원자적 저장, metadata 읽기, diff 파일 생성, 외부 diff 도구 실행을 구현했다.
- 외부 변경 감지는 Tauri file watcher와 polling을 함께 사용한다. watcher는 부모 디렉터리를 감시하고 대상 Markdown 파일 경로만 필터링한다.
- Tauri 파일 접근은 MVP에서 단일 파일 열기/저장부터 시작한다. 작업 폴더/vault 탐색은 후순위다.
- SQLite sidecar는 Markdown 파일 옆의 `{파일명}.mindmap.sqlite`로 시작했다. 현재는 view state key-value 저장소로 사용한다.
- 줌은 문서 변경이 아니라 앱 전용 view state로 취급한다. 툴바 버튼과 `Cmd/Ctrl + +`, `Cmd/Ctrl + -`, `Cmd/Ctrl + 0` 단축키로 조정하며 SQLite view state에 저장된다.
- Tauri dialog plugin 권한을 `src-tauri/capabilities/default.json`에 추가했다.
- 키보드 기본 조작은 아웃라이너에 가깝게 시작했다: 편집 중 Enter 다음 형제 이동/생성, Tab 첫 자식 이동/생성, Shift+Enter 위 형제 이동/생성, Shift+Tab 부모 포커스, Cmd/Option+ArrowUp/Down 이동, Cmd/Option+Backspace 삭제. 선택 모드 Enter는 편집 시작이다.
- 편집 중 `Esc`는 선택 모드로 빠져나오되, 텍스트가 정확히 빈 일반 노드에서는 해당 노드를 삭제한다.
- 키바인딩 도움말을 추가했다. 툴바 `?` 버튼, 선택 모드 `?`, `Cmd/Ctrl + /`로 현재 구현된 단축키를 확인할 수 있다.
- 선택 모드를 추가했다. Esc로 편집 모드에서 빠져나오고, 선택 모드에서 화살표는 화면 좌표 기준으로 위/아래/왼쪽/오른쪽의 가장 가까운 노드를 선택한다.
- 클립보드 subtree 복사/붙여넣기를 추가했다. 복사 내용은 앱 전용 메타데이터 없이 Markdown 목록 텍스트만 사용한다.
- 노드 옆에 상시 노출되던 `+`, `S`, `Del` 액션 버튼을 제거했다. 자식 추가, 형제 추가, 삭제는 키보드 단축키로 수행한다.
- 검증: Vitest 57개 통과, Playwright e2e 18개 통과, TypeScript check 통과, Vite production build 통과.
